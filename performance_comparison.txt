5. RELIABILITY & FAILURE POINTS
===============================================================================

OLD SYSTEM failure points (any one breaks the chain):
  1. ZKBioTime service crashes → devices can't push data
  2. Redis down → Celery workers stop
  3. TAPSync PHP cron fails → data doesn't flow to MOL
  4. Access MDB file locked → COM/ADODB errors
  5. Network between TAPSync and remote MOL DB → sync fails
  6. MySQL connection limit hit (3 apps competing)
  7. COM object registration issues on Windows updates

NEW SYSTEM failure points:
  1. Python server crashes → devices retry automatically (built into protocol)
  2. MySQL down → both server and dashboard affected
  That's it. Two components, two failure points.

Recovery:
  OLD: Must restart multiple services in correct order
  NEW: Single process restart (`python run.py`) recovers everything

CONCLUSION
===============================================================================

The new system eliminates the multi-hop architecture that causes delays and
failures in the old system. By communicating directly with devices via the
PUSH/ADMS protocol (the same protocol ZKBioTime uses internally), it removes
the middleman while adding features the old system lacks: real-time monitoring,
automatic bio sync, device authorization, and background attendance processing.

The primary performance gains come from:
  1. Eliminating 3 intermediate data hops (direct device → DB)
  2. Async I/O instead of synchronous PHP loops
  3. Connection pooling instead of per-request connections
  4. Background workers instead of manual/cron batch processing
  5. Single database instead of 3 databases + Access MDB

Recommended transition: Run both systems in parallel for 1-2 weeks, compare
daily punch counts, then cut over fully once confidence is established.
